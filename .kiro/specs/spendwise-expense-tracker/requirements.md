# Requirements Document

## Introduction

SpendWise is a secure, cloud-based expense tracking system that enables users to record, manage, and analyze their income and expenses in real time across multiple devices. The system provides budgeting tools, visual financial insights, multi-wallet support, custom categories, recurring transactions, and reliable data synchronization to help users gain full visibility and control over their personal finances.

## Glossary

- **SpendWise_System**: The complete expense tracking application including frontend (React Native/Expo), backend (Node.js/Express), and database (PostgreSQL/Neon)
- **Auth_Service**: The authentication module responsible for user registration, login, session management, refresh tokens, and token validation using JWT
- **Transaction_Service**: The module responsible for creating, reading, updating, and deleting income and expense records
- **Budget_Service**: The module responsible for managing user-defined budgets, tracking spending against limits, and issuing alerts
- **Wallet_Service**: The module responsible for managing multiple wallets/accounts, tracking balances, and handling transfers between wallets
- **Category_Service**: The module responsible for managing user-defined income and expense categories with icons and colors
- **Recurring_Transaction_Service**: The module responsible for managing scheduled recurring transactions and auto-generating transactions on due dates
- **Notification_Service**: The module responsible for creating, storing, and managing user notifications including budget alerts
- **Sync_Engine**: The module responsible for synchronizing local data with cloud storage, handling offline caching, and resolving conflicts
- **Analytics_Engine**: The module responsible for generating reports, charts, and financial insights from transaction data
- **User**: A registered individual who interacts with the SpendWise application
- **Transaction**: A single income or expense record containing amount, type, category_id, description, date, and wallet_id
- **Wallet**: A virtual representation of a financial account (e.g., Cash, Bank, GCash, Maya)
- **Category**: A user-defined classification for transactions with a name, type (income or expense), icon, and color
- **Budget**: A user-defined spending limit for a specific category over a defined period (weekly or monthly) with start and end dates
- **Budget_Tracking**: A record that tracks the spent amount against a specific budget
- **Recurring_Transaction**: A scheduled transaction that automatically generates new transactions at a defined interval (daily, weekly, or monthly)
- **Notification**: A message generated by the system to inform the user of budget alerts or other events
- **Session**: A record containing a refresh token associated with a user for maintaining authentication across token expirations
- **JWT**: JSON Web Token used for stateless session authentication

## Requirements

### Requirement 1: User Registration

**User Story:** As a new user, I want to register an account with my email and password, so that I can securely access SpendWise across multiple devices.

#### Acceptance Criteria

1. WHEN a user submits a registration form with a valid name, email, and password, THE Auth_Service SHALL create a new user account and return a JWT access token and refresh token
2. WHEN a user submits a registration form with an email that already exists, THE Auth_Service SHALL reject the registration without creating any account or modifying any data, and return an error message indicating the email is already in use
3. WHEN a user submits a registration form with an invalid email format, THE Auth_Service SHALL reject the registration and return a validation error message
4. WHEN a user submits a registration form with a password shorter than 8 characters, THE Auth_Service SHALL reject the registration and return a validation error message
5. THE Auth_Service SHALL store user passwords as bcrypt hashes with a minimum cost factor of 10
6. WHEN a user is successfully registered, THE Auth_Service SHALL create a session record with the refresh token and its expiration timestamp

### Requirement 2: User Login and Session Management

**User Story:** As a registered user, I want to log in with my credentials and maintain my session, so that I can access my financial data securely without frequent re-authentication.

#### Acceptance Criteria

1. WHEN a user submits valid email and password credentials, THE Auth_Service SHALL authenticate the user and return a JWT access token and refresh token
2. WHEN a user submits invalid credentials, THE Auth_Service SHALL reject the login attempt and return a generic authentication error without revealing which field is incorrect
3. THE Auth_Service SHALL issue JWT access tokens with a maximum expiration time of 15 minutes
4. THE Auth_Service SHALL issue refresh tokens with a maximum expiration time of 7 days
5. WHEN a user authenticates via Google OAuth, THE Auth_Service SHALL create or link the account and return a JWT access token and refresh token regardless of any prior failed login attempts in the same session (Future Scope)
6. IF the Auth_Service receives a request with an expired access token on a protected endpoint, THEN THE Auth_Service SHALL reject the request with a 401 status code
7. WHEN a user submits a valid refresh token, THE Auth_Service SHALL issue a new JWT access token and a new refresh token
8. WHEN a user submits an expired or invalid refresh token, THE Auth_Service SHALL reject the request and require re-authentication
9. WHEN a user logs out, THE Auth_Service SHALL invalidate the session record associated with the refresh token

### Requirement 3: Category Management

**User Story:** As a user, I want to create and manage custom categories for my transactions, so that I can organize my finances in a way that makes sense to me.

#### Acceptance Criteria

1. WHEN a user creates a category with a name, type (income or expense), and optional icon and color, THE Category_Service SHALL create the category record and associate it with the user account
2. WHEN a user creates a category with a name that already exists for the same type under the same user, THE Category_Service SHALL reject the request and return a duplicate error
3. WHEN a user updates a category name, icon, or color, THE Category_Service SHALL save the changes
4. WHEN a user deletes a category that has associated transactions, THE Category_Service SHALL reassign the linked transactions to a default "Uncategorized" category of the same type and then delete the original category
5. WHEN a user requests their category list, THE Category_Service SHALL return all categories belonging to the authenticated user grouped by type
6. THE Category_Service SHALL validate that the type field contains only "income" or "expense"

### Requirement 4: Add Transaction

**User Story:** As a user, I want to add income and expense transactions, so that I can keep an accurate record of my finances.

#### Acceptance Criteria

1. WHEN a user submits a new transaction with amount, type (income or expense), category_id, description, date, and wallet_id, THE Transaction_Service SHALL create the transaction record and associate it with the user account
2. WHEN a user submits a transaction with type "expense", THE Wallet_Service SHALL decrease the associated wallet balance by the transaction amount
3. WHEN a user submits a transaction with type "income", THE Wallet_Service SHALL increase the associated wallet balance by the transaction amount
4. WHEN a user submits a transaction with an amount less than or equal to zero, THE Transaction_Service SHALL reject the transaction and return a validation error
5. WHEN a user submits a transaction without a required field (amount, type, category_id, wallet_id, date), THE Transaction_Service SHALL reject the transaction and return a validation error listing the missing fields
6. WHEN a new expense transaction is created, THE Budget_Service SHALL update the spent amount in the budget_tracking record for any matching category budget in the current period
7. THE Transaction_Service SHALL set the updated_at timestamp to the current time on creation

### Requirement 5: Edit Transaction

**User Story:** As a user, I want to edit existing transactions, so that I can correct mistakes in my financial records.

#### Acceptance Criteria

1. WHEN a user submits an update to an existing transaction, THE Transaction_Service SHALL apply the changes and update the transaction record with a new updated_at timestamp
2. WHEN a user changes the amount or type of a transaction, THE Wallet_Service SHALL recalculate the associated wallet balance to reflect the difference
3. WHEN a user attempts to edit a transaction that does not belong to the user, THE Transaction_Service SHALL reject the request with an authorization error
4. WHEN a user changes the category of an expense transaction, THE Budget_Service SHALL recalculate the spent amounts in budget_tracking for both the old and new category budgets; IF the transaction type is income, THEN THE Budget_Service SHALL NOT recalculate any budget amounts

### Requirement 6: Delete Transaction

**User Story:** As a user, I want to delete transactions, so that I can remove erroneous records from my history.

#### Acceptance Criteria

1. WHEN a user deletes an expense transaction, THE Wallet_Service SHALL increase the associated wallet balance by the deleted transaction amount
2. WHEN a user deletes an income transaction, THE Wallet_Service SHALL decrease the associated wallet balance by the deleted transaction amount
3. WHEN a user attempts to delete a transaction that does not belong to the user, THE Transaction_Service SHALL explicitly reject the request with an authorization error
4. WHEN a user deletes an expense transaction, THE Budget_Service SHALL decrease the spent amount in budget_tracking for the matching category budget in the relevant period

### Requirement 7: Transaction History and Search

**User Story:** As a user, I want to view, filter, and search my transaction history, so that I can find specific records and review my financial activity.

#### Acceptance Criteria

1. WHEN a user requests their transaction history, THE Transaction_Service SHALL return a paginated list of transactions sorted by date in descending order
2. WHEN a user applies a date range filter, THE Transaction_Service SHALL return only transactions within the specified start and end dates
3. WHEN a user applies a category filter, THE Transaction_Service SHALL return only transactions matching the specified category_id
4. WHEN a user applies a type filter (income or expense), THE Transaction_Service SHALL return only transactions matching the specified type
5. WHEN a user submits a search query, THE Transaction_Service SHALL return transactions where the description contains the search term (case-insensitive)
6. THE Transaction_Service SHALL return only transactions belonging to the authenticated user

### Requirement 8: Budget Creation and Management

**User Story:** As a user, I want to set budgets for specific categories, so that I can control my spending and improve financial discipline.

#### Acceptance Criteria

1. WHEN a user creates a budget with a category_id, amount_limit, period (weekly or monthly), and start_date, THE Budget_Service SHALL create the budget record and an associated budget_tracking record with spent initialized to zero
2. WHEN a user creates a budget for a category that already has an active budget in the same period, THE Budget_Service SHALL reject the request and return an error indicating a duplicate budget exists
3. WHEN a user updates a budget amount_limit, THE Budget_Service SHALL save the new limit and recalculate the budget progress percentage
4. WHEN a user deletes a budget, THE Budget_Service SHALL remove the budget record and its associated budget_tracking record
5. THE Budget_Service SHALL calculate budget progress as the percentage of the spent amount (from budget_tracking) relative to the budget amount_limit
6. THE Budget_Service SHALL validate that the period field contains only "weekly" or "monthly"

### Requirement 9: Budget Alerts and Notifications

**User Story:** As a user, I want to receive alerts when I approach or exceed my budget limits, so that I can adjust my spending before overspending.

#### Acceptance Criteria

1. WHEN the spent amount in budget_tracking reaches 50% of the budget amount_limit, THE Notification_Service SHALL create a warning notification for the user with title and message indicating the budget status
2. WHEN the spent amount in budget_tracking reaches 75% of the budget amount_limit, THE Notification_Service SHALL create a caution notification for the user with title and message indicating the budget status
3. WHEN the spent amount in budget_tracking reaches 100% of the budget amount_limit, THE Notification_Service SHALL create a critical notification indicating the budget is fully consumed
4. THE Notification_Service SHALL generate each threshold notification at most once per budget per period
5. WHEN a user requests their notifications, THE Notification_Service SHALL return all notifications for the authenticated user sorted by created_at in descending order
6. WHEN a user marks a notification as read, THE Notification_Service SHALL update the is_read field to true

### Requirement 10: Wallet Management

**User Story:** As a user, I want to manage multiple wallets representing different accounts, so that I can track where my money is held.

#### Acceptance Criteria

1. WHEN a user creates a wallet with a name and initial balance, THE Wallet_Service SHALL create the wallet record and associate it with the user account
2. WHEN a user requests their wallet list, THE Wallet_Service SHALL return all wallets belonging to the authenticated user with current balances
3. WHEN a user updates a wallet name, THE Wallet_Service SHALL save the updated name
4. WHEN a user attempts to delete a wallet, THE Wallet_Service SHALL reject the deletion if the wallet has associated transactions OR a non-zero balance, and return an error indicating the reason for rejection
5. THE Wallet_Service SHALL prevent wallet balances from becoming negative for expense transactions

### Requirement 11: Wallet Transfers

**User Story:** As a user, I want to transfer money between my wallets, so that I can reflect real movements of funds between my accounts.

#### Acceptance Criteria

1. WHEN a user initiates a transfer specifying source wallet, destination wallet, and amount, AND all validation checks pass, THE Wallet_Service SHALL decrease the source wallet balance and increase the destination wallet balance by the transfer amount
2. WHEN a user initiates a transfer with an amount exceeding the source wallet balance, THE Wallet_Service SHALL reject the transfer and return an insufficient funds error
3. WHEN a user initiates a transfer with an amount less than or equal to zero, THE Wallet_Service SHALL reject the transfer and return a validation error
4. WHEN a user initiates a transfer between the same wallet, THE Wallet_Service SHALL reject the transfer and return a validation error
5. THE Wallet_Service SHALL record all transfer attempts (successful and failed) as two linked transactions (expense from source, income to destination) for audit purposes

### Requirement 12: Recurring Transactions [FUTURE SCOPE]

**User Story:** As a user, I want to set up recurring transactions, so that regular income and expenses are automatically recorded without manual entry.

#### Acceptance Criteria

1. WHEN a user creates a recurring transaction with title, amount, type, category_id, wallet_id, interval (daily, weekly, or monthly), and next_due_date, THE Recurring_Transaction_Service SHALL create the recurring transaction record
2. WHEN the current date matches or exceeds the next_due_date of a recurring transaction, THE Recurring_Transaction_Service SHALL generate a new transaction with the specified amount, type, category_id, and wallet_id
3. WHEN a recurring transaction is executed, THE Recurring_Transaction_Service SHALL advance the next_due_date by the specified interval
4. WHEN a user updates a recurring transaction, THE Recurring_Transaction_Service SHALL save the changes and apply the updated values to future generated transactions
5. WHEN a user deletes a recurring transaction, THE Recurring_Transaction_Service SHALL stop generating future transactions but preserve previously generated transactions
6. THE Recurring_Transaction_Service SHALL validate that the interval field contains only the case-sensitive lowercase values "daily", "weekly", or "monthly"

### Requirement 13: Reports and Analytics

**User Story:** As a user, I want to view reports and charts of my financial activity, so that I can understand my spending habits and make informed decisions.

#### Acceptance Criteria

1. WHEN a user requests a monthly summary report, THE Analytics_Engine SHALL return the total income, total expenses, and net balance for the specified month
2. WHEN a user requests a category breakdown, THE Analytics_Engine SHALL return the total spent per category for the specified period
3. WHEN a user requests spending trends, THE Analytics_Engine SHALL return monthly expense totals for the past 6 months
4. THE Analytics_Engine SHALL compute all report data from the transactions belonging to the authenticated user only
5. WHEN a user requests a report for a period with no fully processed transactions, THE Analytics_Engine SHALL return zero values for all metrics instead of an error; pending or cached transactions SHALL NOT be included in report calculations

### Requirement 14: Data Synchronization [FUTURE SCOPE]

**User Story:** As a user, I want my data to sync automatically across devices, so that I always see up-to-date information regardless of which device I use.

#### Acceptance Criteria

1. WHEN the device has network connectivity, THE Sync_Engine SHALL synchronize local changes to the cloud within 5 seconds of a data modification
2. WHILE the device has no network connectivity, THE Sync_Engine SHALL have its caching mechanism active and ready to cache all data modifications locally
3. WHEN network connectivity is restored, THE Sync_Engine SHALL synchronize all cached local changes to the cloud in chronological order
4. IF a sync conflict occurs where the same record was modified on multiple devices, THEN THE Sync_Engine SHALL resolve the conflict using a last-write-wins strategy based on the updated_at timestamp
5. THE Sync_Engine SHALL preserve data integrity by using database transactions for all synchronization operations

### Requirement 15: Offline Support [FUTURE SCOPE]

**User Story:** As a user, I want to continue using the app when offline, so that I can record transactions even without internet access.

#### Acceptance Criteria

1. WHILE the device has no network connectivity, THE SpendWise_System SHALL allow the user to create, edit, and delete transactions using locally cached data
2. WHILE the device has no network connectivity, THE SpendWise_System SHALL display the most recently synced wallet balances and transaction history
3. WHEN the device transitions from offline to online, THE Sync_Engine SHALL display a synchronization status indicator to the user
4. THE Sync_Engine SHALL store a maximum of 1000 pending offline changes before warning the user to connect to sync

### Requirement 16: API Performance

**User Story:** As a user, I want the app to respond quickly, so that I can manage my finances without frustrating delays.

#### Acceptance Criteria

1. THE SpendWise_System SHALL return API responses within 2 seconds for 95% of requests under normal load
2. THE SpendWise_System SHALL return API responses within 3 seconds for 99% of requests under normal load
3. WHEN the transaction history reaches 500 or more records, THE Transaction_Service SHALL use pagination with a default page size of 20 records
4. THE SpendWise_System SHALL support at least 100 concurrent authenticated users without performance degradation

### Requirement 17: Data Security

**User Story:** As a user, I want my financial data to be secure, so that I can trust the app with sensitive information.

#### Acceptance Criteria

1. THE Auth_Service SHALL transmit all data over HTTPS with TLS 1.2 or higher
2. THE Auth_Service SHALL store passwords using bcrypt hashing with a minimum cost factor of 10
3. THE SpendWise_System SHALL encrypt sensitive user data at rest in the database (Future Scope)
4. WHEN a user session refresh token expires after 7 days, THE Auth_Service SHALL require re-authentication
5. THE SpendWise_System SHALL log all authentication attempts with timestamp, IP address, and success status for security auditing, independent of session expiry configuration

### Requirement 18: Transaction Data Parsing

**User Story:** As a developer, I want transaction data to be reliably serialized and deserialized, so that data integrity is maintained during sync and API communication.

#### Acceptance Criteria

1. THE Transaction_Service SHALL serialize transaction objects to JSON format for API responses
2. THE Transaction_Service SHALL deserialize incoming JSON payloads into validated transaction objects
3. FOR ALL valid Transaction objects, serializing then deserializing SHALL produce an equivalent Transaction object (round-trip property)
4. WHEN the Transaction_Service receives syntactically malformed JSON, THE Transaction_Service SHALL return a descriptive parse error with the location of the syntax issue; semantic validation of transaction object fields SHALL be handled separately from JSON parse errors
